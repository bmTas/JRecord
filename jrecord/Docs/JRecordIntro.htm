    Introduction

    This document covers the basic JRecord Operation using the IOBuilder interface, Cobol2Csv / Csv2Cobol programs and Cobol2Xml / Xml2Cobol programs. For most user's the IOBuilder interface is the easiest way to use JRecord. For those writing utilities/need maximum flexibility you may need to look at Classic JRecord Basics and "Cobol Item Schema Details" as well.

    Basically you can read/write Cobol data files by:

    • "Cobol2Csv & Csv2Cobol programs" - Convert Cobol Data files to / from Csv files. They are useful for processing simple Cobol Data files.
    • "Cobol to/from Xml" - Convert Cobol Files to/from Xml.
    • "Cobol to Json" - Convert cobol Files to JSon
    • JRecord "IO Builders" - A simple interface to JRecord that allows access to complicated Cobol Data files (multi record, binary ??, duplicate field names etc).
    • Classic JRecord - Older more complicated interface.
    • "RecordEditor Cobol Import" - In the RecordEditor Cobol import, you can specify both a Cobol Copybook and a sample Cobol Data file. The RecordEditor will try and work out the appropriate attributes for the file. You then have the option of generating a JRecord IOBuilder for the file. This should be useful for new users.
    • "JRecord Code Generation" - Can be used to generate Skelton Java~JRecord programs from a Cobol Copybook.

    The Cobol2Csv and Cobol2Xml interfaces are useful because

    • Cobol Interface is isolated to 2 programs.
    • Any language that supports Csv/Xml can be used.
    • They allows Cobol programmers to talk Cobol and the Java/Python/... programmers to work with files Formats (Csv / Xml) that they are confortable with.

    IO Builders

    Creating a Cobol IOBuilder

    The easiest way to create an IO Builder is via the JRecordInterface1 class: 

    20             ICobolIOBuilder iob = JRecordInterface1.COBOL
42                                         .newIOBuilder(copybookName)
43                                             .setFont("cp037")          // US EBCDIC
44                                             .setFileOrganization(Constants.IO_FIXED_LENGTH_RECORDS);

    The methods available for Cobol IO Builder include

    • setFont - set Font or Characterset
    • setDialect - specify the Cobol dialect (Default=Mainframe)
    • setFileOrganization - Specify how the Records are organised in the file.

    Generatng IOBuilder Code

    To make it easier to get started using JRecord IOBuilders, there 3 code generation options:

    JRecord CodeGen

    The CodeGen program will produce Java~JRecord from a Cobol Copybook. See "JRecord Code Generation" for more details

    RecordEdit CodeGen

    The CodeGen program is now included in the RecordEditor, it is available under the Generate option. See RecordEditor Generate documentation for more details

    RecordEditor Cobol Import

    In the RecordEditor Cobol Import dialog, you can enter both a Cobol Copybook and a Cobol Data File. The RecordEditor Cobol Import dialog will try and work the file attributes for you. Press the Generate JREcord Code and the system will generate 2 files:

    • A sample JRecord IOBuilder
    • A class that defines the Cobol field names as constants. You can then use Code Completion in you IDE rather than entering the Cobol names.

    This option will be usefull to new users who are unsure of what attributes to use in IOBuilder's. See "RecordEditor Cobol Import" for more details

    There are JavaDoc descriptions for the IOBuilder methods. The JavaDoc will explain the options for the method. In Eclipse you should see:

    setFileOrganization

    In the cobol world, files are not always the cr delimited Text files you are familiar with in the Windows / *nix world. In Cobol, files can be

    Fixed-Width

    Every Record (line) is the same Length, there are no cr/lf Record terminators.

    Variable Length

    Each Record is precedded by the record length (2/4 byte binary integer). There is no single VB Each Cobol compiler implements it differently.

    The setFileOrganization method lets you define File-Organisation (or File Structure) of the Cobol data file.

    JavaDoc details

    Typical values are:

    Constants.IO_FIXED_LENGTH

    Fixed Length Records (no cr/lf) using an 8 bit characterset (e.g. ascii). It could be binary

    Constants.IO_FIXED_LENGTH_CHAR

    Fixed Length Character Records (no cr/lf). Typically used for utf-8 / utf-16 files.

    Constants.IO_VB

    IBM Mainframe VB file (each record is preceded by a Record-Length

    Constants.IO_VB_DUMP

    IBM Mainframe VB file with both the Block-Lengths and Record-Lengths included

    Xml Schema

    In JRecord, you can also use a Xml Schema instead of a Cobol Copybook to define a file.

    A sample Fixed-Width Xml schema: 

     2 <RECORD RECORDNAME="DTAR020" COPYBOOK="DTAR020" DELIMITER="&lt;Tab&gt;" 
 3         FONTNAME="CP037" FILESTRUCTURE="Default" STYLE="0" RECORDTYPE="RecordLayout"
 4         LIST="Y" QUOTE="" RecSep="default">
 5         <FIELDS>
 6                 <FIELD NAME="KEYCODE-NO" POSITION="1"  LENGTH="8" TYPE="Char" />
 7                 <FIELD NAME="STORE-NO"   POSITION="9"  LENGTH="2" TYPE="Mainframe Packed Decimal (comp-3)" />
 8                 <FIELD NAME="DATE"       POSITION="11" LENGTH="4" TYPE="Mainframe Packed Decimal (comp-3)" />
 9                 <FIELD NAME="DEPT-NO"    POSITION="15" LENGTH="2" TYPE="Mainframe Packed Decimal (comp-3)" />
10                 <FIELD NAME="QTY-SOLD"   POSITION="17" LENGTH="5" TYPE="Mainframe Packed Decimal (comp-3)" />
11                 <FIELD NAME="SALE-PRICE" POSITION="22" LENGTH="6" DECIMAL="2" TYPE="Mainframe Packed Decimal (comp-3)" />
12         </FIELDS>
13 </RECORD>

    You can define a Csv Xml-Schema: 

       1 <?xml version="1.0" encoding="UTF-8"?>
   2 <RECORD RECORDNAME="" COPYBOOK="" DELIMITER=";" FILESTRUCTURE="CSV_NAME_1ST_LINE" STYLE="0" RECORDTYPE="Delimited" LIST="N" QUOTE="&quot;" RecSep="default" LINE_NO_FIELD_NAMES="1">
   3         <FIELDS>
   4                 <FIELD NAME="Sku"   POSITION="1" TYPE="Char"/>
   5                 <FIELD NAME="Store" POSITION="2" TYPE="NumAnyDecimal"/>
   6                 <FIELD NAME="Date"  POSITION="3" TYPE="NumAnyDecimal"/>
   7                 <FIELD NAME="Dept"  POSITION="4" TYPE="NumAnyDecimal"/>
   8                 <FIELD NAME="Qty"   POSITION="5" TYPE="NumAnyDecimal"/>
   9                 <FIELD NAME="Price" POSITION="6" TYPE="NumAnyDecimal"/>
  10                 <FIELD NAME="GST"   POSITION="7" TYPE="NumAnyDecimal"/>
  11         </FIELDS>
  12 </RECORD>

    You then use JRecordInterface1.SCHEMA_XML to create an IOBuilder from an Xml-Schema: 

    5      IIOBuilder ioBldr = JRecordInterface1.SCHEMA_XML
6                                           .newIOBuilder(copybookName);

    You then create readers / writers as you would with Cobol IOBuilder. Using the Xml-IOBuilder to create readers/writers is exactly the same as with a Cobol IOBuilder. See "Reading a simple Binary EBCDIC file" for examples of using a Cobol IOBuilder

    Cobol2Csv & Csv2Cobol programs

    The Cobol2Csv and Csv2Cobol are now in a separate project - CobolToCsv. You can download the programs / documentation from CobolToCsv.

    The RecordEditor can now generate Cobol2Csv bat/shell scripts, see RecordEditor - Generate CobolToXml

    If you have simple COBOL files (one record type and no duplicate field names), you can use Cobol2Csv and Csv2Cobol programs. These programs will convert a single record cobol file to / from a CSV file. If you pro The big advantages are:

    • No programming is involved.
    • You keep your code base clear of Cobol, you just using existing Csv Skills / packages.
    • Can use other languages (other than java) or load into a DB.

    It make sense to use Cobol2Csv and Csv2Cobol if

    1. Files are Simple one record files with no duplicate field names.
    2. You are not going to process the files in Java (i.e. plan to process the files in a language other than java or you are going to directly load the files into a database).
    3. Your organisation already uses a package other than JRecord to read CSV files.
    4. You want your business application clear of **Cobol** references.

    Note: It makes less sense to convert the files to CSV if

    • You use JRecord to read / write the CSV files. You might as well just read / write the raw Cobol files with JRecord.
    • If you do not use an existing CSV package, you might as well just learn JRecord.


    Cobol2Csv examples: 

       Basically to call Cobol2Csv / Csv2Cobol
     java -jar Cobol2Csv <program arguments>
 
     java -jar Csv2Cobol <program arguments>
 
   or
 
     Cobol2Csv.bat/sh <program arguments>
 
     Csv2Cobol.bat/sh <program arguments>
 
     ------------------------------------------------------------------------------
 
     Copy a fixed length Mainframe file to a Tab delimited File 
     java -jar ../lib/Cobol2Csv.jar \
                  -C DTAR020.cbl  -FS Fixed_Length    -IC CP037 \
                  -Q DoubleQuote  -D tab \
                  -I in/DTAR020.bin  -O out/o_DTAR020_a.csv
 
     Copy a Line-Sequential (ordinary Text) file to a : delimited File
     with _'s instead of -'s in the column names 
     java -jar ../lib/Cobol2Csv.jar  -C Test2.cbl  -FS Text \
                  -D : -Rename Change_Minus_To_Undescore \
                  -I in/Test2.txt  -O out/o_Test2-rb.csv
 
     Convert a Csv file back into a Fixed Width Mainframe Cobol Data file
     java -jar ../lib/Csv2Cobol.jar ^
                               -C G:/Users/Bruce01/RecordEditor_HSQL/Copybook/Cobol/DTAR020.cbl ^
                               -Delimiter , ^
                               -OFS Fixed_Length ^
                               -OC cp037 ^
                               -I out/DTAR020.bin.csv ^
                               -O out/DTAR020.fromCsv.bin

    Cobol2Csv, Csv2Cobol program arguments: 

      -I  or -InputFile           : Input file
  -O  or -OutputFile          : Output file
  -C  or -Copybook            : Cobol Copybook file
  -D  or -Delimiter           : Field Seperator (space, tab or value)
  -Q  or -Quote               : Quote, you can use DoubleQuote SingleQuote as well as an a the char
  -IC or -InputCharacterSet   : Input  font or character set
  -OC or -OutputCharacterSet  : Output font or character set
  -IFS or -InputFileStructure : Input File Structure:
  -OFS or -OutputFileStructure: Output File Structure:
      Default      : Determine by Copybook
      Text         : Use Standard Text IO
      Fixed_Length : Fixed record Length binary
      Mainframe_VB : Mainframe VB File
      Mainframe_VB_As_RECFMU : Mainframe VB File including BDW (block descriptor word)
      FUJITSU_VB   : Fujitsu Cobol VB File
      Open_Cobol_VB: Gnu Cobol VB File
  -Dialect : Cobol Dialect
      0 or Intel        : Intel little endian
      1 or Mainframe    : Mainframe big endian (Default)
      4 or Open_Cobol_Little_Endian_(Intel)  : Gnu:Cobol 
      2 or Fujitsu      : Fujitsu Cobol
  -Rename         : How to update cobol variable names
      0 or Leave_Asis                : Use the COBOL name
      1 or Change_Minus_To_Underscore: Change '-(),' to '_'
      2 or Drop_Minus                : Drop minus ('-') from the name
  -CsvParser : Controls how Csv fields are written / parsed
      0 or Basic_Parser                          : Parse Csv - when a field starts with " look for "<FieldSeparator>or "<eol&gt
      1 or Standard_Parser                       : Parse CSV matching Quotes
      2 or Standard_Parse_Quote_4_Char_Fields    : Standard Parser, add Quotes to all Char fields
      3 or Basic_Parser_Column_names_in_quotes   : Basic Parser, Field (Column) names in Quotes
      4 or Standard_Parser_Column_names_in_quotes: Standard Parser, Field (Column) names in Quotes
      5 or Standard_Parser_Quote_4_Char_Fields_Column_names_in_quotes        : Standard Parser, Char fields in Quotes,  Field (Column) names in Quotes
      6 or Basic_Parser_Delimiter_all_fields     : Basic Parser, Field Separator for all fields
      7 or Basic_Parser_Delimiter_all_fields+1   : Basic Parser, Field Separator for all fields + extra seperator at the End-of-Line

    The Cobol2Xml and Xml2Cobol are now in a separate project - CobolToXml. You can download the programs / documentation from CobolToXml.

    The programs Data2Xml and Xml2Data will convert Cobol Data files to/from Xml. It makes sense to use this interface if:

-input	- Input file
-output	- Output file

-dropCopybookName	- (true/false) wether to drop the cobol copybook name from the start of the Xml Tags
-mainXmlTag     	- The outermost Xml tag to use (default coboldata)
-tagFormat     	- How Cobol Variable names are reformated to Xml tags:
Asis    	- Use the Cobol Variable name
Underscore	- Convert - to _,         COBOL-VAR-NAME ==> COBOL_VAR_NAME
CamelCase	- Convert to Camel Case,  COBOL-VAR-NAME ==> cobolVarName
-fileOrganisation	- "file organization" of the Cobol data file
	Text    	- Standard Windows/Unix text file (single byte character-set)
	FixedWidth	- File where lines (records) are the same length no \n
	Mainframe_VB	- Mainframe VB, file consists of <record-length><record-data>
	GNUCobol_VB	- GNU Cobol VB, file consists of <record-length><record-data>
-dialect	- Cobol Dialect
	Mainframe	- Mainframe cobol
	Futjitsu	- Fujitsu PC cobol
	GNUCobol	- GNU Cobol (little endian, ie intel)
	GNUCobolBE	- GNU Cobol (big endian, ie IBM, Sun(oracle))
-split	- Option for Splitting Copybook into seperate records
	None   	- No Split
	01     	- Split on 01
	Highest	- On Highest Repeating
-recordSelection	- Record Selection, can be used multiple time,
format: -recordSelection RecordName field=value
-recordParent   	- Record Parent, can be used multiple time,
format: -recordParent    RecordName ParentRecord
